# ハーネスエンジニアリング ベストプラクティス

このドキュメントはClaude Codeのハーネスエンジニアリング（Claude Codeの動作を制御・最適化する設計手法）のベストプラクティスをまとめたものです。新しいセッションや新しいプロジェクトでこのプラクティスに従って開発を進めてください。

## 3大原則

### 1. コンテキストは最も希少なリソース

CLAUDE.mdのロード、スキルの展開、ファイルの読み込み — 全てがコンテキストを消費する。無駄な情報をコンテキストに入れない設計が最も重要。

### 2. 具体性（Specificity）で信頼性向上

「きちんとやれ」ではなく「PBR < 1.0, ROE > 15%, 配当 > 3%で抽出」と書く。曖昧な指示はClaudeの任意判定を招き、セッションごとに結果がブレる。

### 3. 決定論的制御（Determinism）で確実性確保

| 制御手段 | 強制力 | 用途 |
|---|---|---|
| CLAUDE.md / Skills | 指導（Claudeが従う） | 方針・手順の定義 |
| Hooks | 強制（従わない選択肢なし） | コミット検証、危険操作の阻止 |
| ADR | 判断基準 | CLAUDE.md / Hooksの根拠 |

## 1. CLAUDE.md の書き方

### 必須ルール

- **200行以下を目標。** 超過分はロードされないリスクがある（先頭200行 or 25KBのいずれか）
- 1行ずつ「これがなくてもClaudeは正しく動作するか？」と問う。不要なら削除か `.claude/rules/` に移す
- 見出し（markdown headers）と箇条書きで論理的にグループ化

### 良い例・悪い例

```markdown
# ❌ 悪い例（曖昧）
コードをきれいに書いてください。テストも忘れずに。

# ✅ 良い例（具体的）
- インデント: 2スペース
- テスト: 変更したファイルに対応する *.test.ts を更新
- コミット前: `npm test` を実行し、全テスト通過を確認
```

### 分割の指針

CLAUDE.mdが200行を超えたら `.claude/rules/` に分割する:

```text
CLAUDE.md                          # プロジェクト概要・最重要ルールのみ（100行以下）
.claude/rules/
├── architecture-guard.md          # アーキテクチャ制約
├── commit-review.md               # コミット前チェック
├── data-sources.md                # データ取得方法
├── stock-types.md                 # 銘柄タイプルール
└── indicators.md                  # 監視指標の閾値
```

### 一貫性チェック

複数のCLAUDE.md・ルール・スキル間で矛盾がないか定期レビューする。矛盾があるとClaudeが任意に一方を選択し、予測不能な動作を招く。

## 2. ルールファイル（.claude/rules/）

### パス固有ルール

特定のファイルを扱う時だけロードされるルールを書ける:

```yaml
---
paths:
  - "personas/*.md"
  - "config/*.yaml"
---
```

これにより無関係なファイル作業時のコンテキスト消費を防げる。

### スコープの優先順位

1. `~/.claude/rules/` — ユーザーグローバル（最低優先）
2. `.claude/rules/` — プロジェクトレベル（高優先）
3. `packages/sub/.claude/rules/` — ネストされたルール（最高優先）

### 粒度の指針

- 1ファイル = 1つの関心事
- 「いつ適用するか」が明確なルール
- 曖昧な「心がけ」ではなく、検証可能な条件

## 3. メモリ（Memory）

### CLAUDE.md vs Memory の使い分け

| 用途 | CLAUDE.md | Memory |
|---|---|---|
| 不変のコード規約 | ✅ チェックイン | ❌ |
| ビルドコマンド | ✅ | ❌ |
| ユーザーの好み・スタイル | ❌ | ✅ |
| 過去のセッションで学んだこと | ❌ | ✅ |
| デバッグパターン | ❌ | ✅ |

### メモリファイルの構成

```text
~/.claude/projects/<project>/memory/
├── MEMORY.md              # インデックス（先頭200行がロード）
├── user_profile.md        # ユーザーの役割・知識レベル
├── feedback_*.md          # フィードバック（修正・確認）
├── project_*.md           # プロジェクト状況
└── reference_*.md         # 外部リソースへのポインタ
```

### メモリの書き方

```markdown
---
name: SQL実行前にスキーマ確認
description: findata等のSQLを書く前に必ずカラム構造を確認してからクエリを実行する
type: feedback
---

SQL実行前に必ず対象テーブルのカラム構造を確認してからクエリを書く。

**Why:** `short_ratios` にcodeカラムがないのに `sr.code` を参照してエラーを繰り返した。

**How to apply:** 初めてアクセスするテーブルは `information_schema.columns` で構造を確認してからSQLを組み立てる。
```

### メモリの昇格ルール

- 同じフィードバックが3回出たら → CLAUDE.md またはルールに昇格
- 古くなった情報 → 削除または更新
- MEMORY.mdのインデックスは200行以内に収める

## 4. スキル（Skills）

### スキルの粒度

- 1スキル = 1つのワークフロー
- 大きすぎるスキルは分割する
- 関連する参照ファイルは `reference/` サブディレクトリに置く

### frontmatter の活用

```yaml
---
name: analyze-stock
description: 全ペルソナで個別銘柄分析。テクニカル+ファンダメンタル。$ARGUMENTS
allowed-tools:
  - Read
  - Glob
  - Grep
  - Agent
  - mcp__findata__execute_sql
  - mcp__mcp-gemini-cli__chat
---
```

| フィールド | 用途 |
|---|---|
| `description` | Claudeが自動で呼び出すかどうかの判断材料 |
| `allowed-tools` | 使えるツールを制限（安全性向上） |
| `context: fork` | 隔離サブエージェントで実行（コンテキスト節約） |

### Supporting Files パターン

```text
.claude/skills/analyze-stock/
├── SKILL.md                    # メイン指示
└── reference/
    ├── financial-metrics.md    # 指標計算式の定義
    └── sql-templates.md        # よく使うSQLテンプレート
```

SKILL.md内から `reference/financial-metrics.md を参照` と書く。

## 5. MCPサーバー

### 設定の原則

- **最小限のサーバー数。** 接続が増えるほど不安定になる
- HTTP MCPはホットリロード可能（再起動不要）
- READ ONLYユーザーでDB接続（SQLインジェクション対策の最終防衛線）

### .mcp.json の書き方

```json
{
  "mcpServers": {
    "findata": {
      "type": "http",
      "url": "http://192.168.0.20:3000/mcp"
    },
    "mcp-gemini-cli": {
      "command": "npx",
      "args": ["mcp-gemini-cli", "--allow-npx"]
    }
  }
}
```

### MCPサーバーの選定基準

- **データ取得:** PostgreSQL等のDB → MCP化して `execute_sql` を提供
- **外部API:** ウェブ検索、AI呼び出し → 既存のnpxパッケージを活用
- **計算処理:** できる限りDB側で事前計算。LLMに計算させない

## 6. Hooks

### Hook の3段階設計

| 段階 | 強制力 | 例 |
|---|---|---|
| **Deterministic** | 絶対実行 | フォーマットチェック、ファイル保護 |
| **Decision-based** | 条件判定 | コミットメッセージ検証 |
| **LLM-based** | AI判断 | タスク完了判定 |

### よく使うパターン

```json
{
  "hooks": {
    "Notification": [
      { "command": "afplay /System/Library/Sounds/Glass.aiff" }
    ],
    "Stop": [
      { "command": "afplay /System/Library/Sounds/Glass.aiff" }
    ],
    "PostCompact": [
      { "command": "echo 'コンテキスト圧縮が発生しました。新しいセッションへの切り替えを検討してください。'" }
    ]
  }
}
```

### Exit Code の意味

- **0:** 成功。JSON出力で構造化制御が可能
- **2:** ブロック。stderrがClaudeへのフィードバックになる
- **その他:** 非ブロック。verboseモードでのみ表示

## 7. ADR（Architecture Decision Record）

### ADRを書くタイミング

- 技術的な選択肢が複数あり、1つを選んだ時
- 「なぜこうしたのか」を将来の自分やチームに説明する必要がある時
- 既存のADRを覆す決定をした時（旧ADRをDeprecatedにして新ADRを作成）

### ADRのフォーマット

```markdown
# ADR-XXXX: タイトル

## Status
Accepted / Deprecated / Updated

## Context
なぜこの決定が必要になったか

## Decision
何を決めたか

## Consequences
### Positive
- メリット

### Negative
- デメリット・トレードオフ

## References
- 関連ADR、ドキュメント
```

### ADRとCLAUDE.mdの連携

- CLAUDE.mdのルールの根拠はADRに書く
- `.claude/rules/architecture-guard.md` でADR違反を検知するルールを定義
- コミット前レビューでADR整合性チェックを必須化

## 8. ペルソナシステム

### 設計パターン

| パターン | 実装 | コンテキスト影響 |
|---|---|---|
| **同一コンテキスト** | Claude自身がロール切替 | 大（全ペルソナ分のプロンプト消費） |
| **サブエージェント** | Agent toolで隔離 | 小（メインに要約のみ返る） |
| **外部AI** | Gemini MCP経由 | 最小（別AIのコンテキスト） |

### 推奨構成

```text
personas/
├── quant.md       # Claude自身（テクニカル。データ処理が多い）
├── bull.md        # Gemini経由（ウェブ検索で強気材料探索）
├── bear.md        # Agent隔離（リスク分析。コンテキスト分離）
├── contrarian.md  # Agent隔離（逆張り判定）
├── value.md       # Gemini経由（バリュエーション分析）
├── flow.md        # Claude自身（需給データはDB直結）
├── macro.md       # Gemini経由（マクロはウェブ検索必須）
└── synthesizer.md # Claude自身（統合判断）
```

### ペルソナプロンプトの書き方

```markdown
# Bear（リスク分析担当）

## 役割
ポートフォリオの弱点を容赦なく突き、最悪シナリオを想定する。
「大丈夫だろう」という楽観を排除する。

## データソース
- findata.technicals: テクニカルサマリー（Quantから受け取る）
- findata.margin_interest: 信用残の推移
- Gemini検索: 地政学リスク、規制リスク

## 出力フォーマット
- リスク要因を重要度順にリスト化
- 各リスクに対する撤退条件を明示
- 「何が起きたらSELLか」を具体的な数値で示す
```

## 9. フィードバックループ

### サイクル

```text
1. Claude の行動を観察
2. 「何がうまくいったか」「何が失敗したか」を特定
3. Memory にフィードバックを保存
4. 次のセッションで改善を確認
5. 3回以上同じパターン → CLAUDE.md / ルールに昇格
```

### フィードバックの種類

| 種類 | 保存先 | 例 |
|---|---|---|
| **修正（correction）** | feedback_*.md | 「SQL実行前にスキーマ確認して」 |
| **確認（confirmation）** | feedback_*.md | 「半分損切りの判断は正解だった」 |
| **好み（preference）** | user_*.md | 「Gemini出力は整形して提示して」 |
| **プロジェクト状況** | project_*.md | 「findata移行完了」 |

### 修正と確認の両方を記録する

修正だけ記録すると、うまくいったアプローチを忘れて過度に慎重になる。確認も記録して、良いパターンを強化する。

## 10. コンテキスト管理

### コンテキスト節約テクニック

| テクニック | 効果 | 実装 |
|---|---|---|
| サブエージェントで調査を分離 | 大 | `Agent` tool でExplore/研究を隔離 |
| スキルのオンデマンドロード | 中 | スキルは呼ばれた時だけ展開 |
| ルールのパス指定 | 中 | 関係ないルールをロードしない |
| メモリの活用 | 大 | セッション間で知識を継承 |
| `/compact` で手動圧縮 | 中 | 話題が変わった時に実行 |

### 長いセッションの対処

- 420k tokens（1Mの42%）を超えたら新しいセッションを提案
- 圧縮が発生したらセッション切り替えを推奨
- 重要な決定事項はメモリに保存してから切り替え

### PostCompact Hook

```json
{
  "hooks": {
    "PostCompact": [{
      "command": "echo 'コンテキスト圧縮が発生しました。重要な情報が失われている可能性があります。新しいセッションへの切り替えを検討してください。'"
    }]
  }
}
```

## 11. 実戦から得たプラクティス

このプロジェクト（investment-ai）の運用で確立した、汎用的に使えるプラクティス。

### データ品質

- **SQL実行前にスキーマを確認する。** 初めてアクセスするテーブルは `information_schema.columns` でカラム構造を確認してからクエリを書く。思い込みでカラム名を書くとエラーを繰り返す
- **計算値は外部ソースでクロスチェックする。** DB計算値（配当利回り等）を株探・Yahooファイナンス等と照合して精度を検証する。分割調整漏れ等のバグを早期発見できる
- **外部AI（Gemini等）の出力は信頼しすぎない。** 先物データ等は不正確な場合がある。重要な数値は複数ソースで確認する

### 外部AIとの連携

- **外部AI出力は整形してから提示する。** Geminiの生出力をそのまま貼らない。内容はそのまま維持し、フォーマットだけClaude側で統一して整形する
- **ペルソナ投票制で意思決定する。** 全ペルソナの意見を集約し、多数決の結果をSynthesizerが統合判断する。1つのペルソナの意見だけで判断しない
- **ペルソナは並列実行する。** Bull/Bear/Macro等は依存関係がないので、Agent toolとGeminiで同時に起動して時間を節約する

### ログ・記録

- **スクリーニング結果は必ずログに保存する。** 次回実行時に前回との差分比較ができる。新規候補・消えた候補を追跡する
- **日別ログフォルダ（`logs/YYYY-MM/DD/`）を使う。** 月別だと1フォルダにファイルが溜まりすぎる。1日分のレポートが一箇所にまとまる
- **トレード記録は根拠も書く。** 「なぜ売ったか」「ペルソナの投票結果」「教訓」を含める。次回の同じ状況で参照できる

### コミット

- **論理的な単位でコミットを分割する。** 削除・ドキュメント更新・トレード記録は別コミットにする。「リファクタリング+新機能」を混ぜない
- **markdownlint等のpre-commitフックを必ず通す。** `--no-verify` でのスキップは禁止。エラーが出たら修正してから再コミット
- **コミットメッセージは日本語で、type prefixを付ける。** `feat:`, `fix:`, `docs:`, `refactor:`, `config:`, `chore:` で始める

### 画像活用

- **スクリーンショットを活用する。** API検索より画像貼り付けの方が正確で速い場合がある。Claude Codeは画像を読み取れる
- **外部ツールのスクショからデータを読み取る。** ダッシュボード、グラフ、表のスクショを貼って分析に活用する

### 意思決定プロセス

- **複数の視点（ペルソナ）で判断する。** 1つの視点だけでなく、楽観・悲観・逆張り等の複数視点を並行で分析し、投票制で意思決定する。偏りを防げる
- **判定基準は数値で定義する。** 曖昧な「そろそろ」ではなく「4つの条件のうち3つを満たしたら実行」のように定量化する
- **段階的にアクションする。** 一括ではなく3〜4段階に分けて実行する。全弾を一度に使い切らない設計にする

### 誤検知・誤判定への対策

- **時期的な特殊要因を考慮する。** 定期イベント（決算期、年度末、権利確定日等）の前後はデータの意味が変わることがある。検知ロジックにフィルタを入れるか、結果に注記を付ける
- **DB計算値を外部ソースでクロスチェックする。** 事前計算した値が本当に正しいか、別のデータソースと照合する習慣をつける。特に新しいテーブルやロジック変更後は必須

## チェックリスト

新しいプロジェクトやセッションで以下を確認:

### セットアップ時

- [ ] CLAUDE.md は200行以下か
- [ ] `.claude/rules/` で関心事を分離しているか
- [ ] MEMORY.md のインデックスは200行以内か
- [ ] MCPサーバーは最小限か
- [ ] ADRで重要な決定を記録しているか

### 開発中

- [ ] フィードバック（修正・確認）をメモリに保存しているか
- [ ] スキルの内部が最新のアーキテクチャと整合しているか
- [ ] 旧ツール・旧APIの参照が残っていないか
- [ ] コンテキスト使用量を意識しているか

### セッション終了時

- [ ] 重要な決定事項をメモリ or ADRに記録したか
- [ ] 次のセッションに引き継ぐべき情報があるか
- [ ] CLAUDE.md / ルール / スキルに更新が必要か
